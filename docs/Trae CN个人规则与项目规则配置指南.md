# 一、核心概念解析（官方权威版）

## 1.1 个人规则 (Personal Rules)

官方定义：基于个人使用习惯和需求为AI定制的规则，旨在让AI的输出更符合用户的个性化要求，减少重复沟通成本，固化个人工作风格。

**生效范围**：全局生效，适用于所有项目，不受项目切换影响

**核心价值**：省去重复说明偏好、风格、要求的步骤，让AI快速适配个人工作习惯，提升单人开发效率

**存储位置**：

- Windows: `%USERPROFILE%\.trae\personal_rules.md`

- macOS/Linux: `~/.trae/personal_rules.md`

**官方推荐应用场景**：

- 语言风格：偏好简洁/严谨/幽默等表达方式，固定使用中文或英文交互

- 操作系统：优先提供Windows、macOS或Linux专属解决方案

- 内容深度：控制回答篇幅，选择仅结论、带步骤、附原理、配示例等模式

- 交互方式：直接给出答案、引导式提问补充信息、分步执行任务

- 编码习惯：固定编程语言偏好、注释风格、代码结构、缩进格式

- 个性化偏好：报错处理方式、文档格式、工具使用优先级

## 1.2 项目规则 (Project Rules)

官方定义：针对当前项目的专用规范，绑定项目目录，确保所有团队成员、AI助手遵循统一的编码标准和协作流程，保障团队输出一致性。

**生效范围**：仅当前项目生效，切换项目后自动失效，切换回本项目自动加载

**核心价值**：统一团队开发规范，消除个人习惯差异，降低协作沟通成本，保障代码质量，便于项目维护和交接

**存储位置**：`项目根目录/.trae/rules/project_rules.md`

**官方推荐应用场景**：

- 项目基础信息：项目介绍、技术栈、运行命令、部署方式（强制生效）

- 代码风格规范：缩进、命名法、引号类型、代码格式

- 技术栈约束：框架版本、依赖库、编程语言、禁用语法

- 项目架构规范：目录结构、文件命名、模块划分

- 文档标准：注释格式、README模板、接口文档规范

- 提交规范：Git提交信息、分支管理、代码审核要求

- 安全合规：代码安全要求、漏洞防护、权限管控

# 二、个人规则配置指南（官方操作+社区实操）

## 2.1 官方配置步骤

1. 打开 Trae CN，点击右侧 AI 功能管理**设置按钮** → **规则面板**

2. 在**个人规则**区域点击**+ 创建 personal_rules.md**

3. 系统自动创建文件并在编辑器中打开，支持Markdown和纯文本编写

4. 输入规则内容，语句尽量简洁明确，避免歧义、模糊表述

5. 保存文件，规则立即生效；建议开启全新对话，让规则完全加载

## 2.2 个人规则修改与关闭

- 修改规则：直接打开个人规则文件编辑，保存后即刻生效

- 临时关闭：清空文件内容保存，即可关闭个人规则

- 备份迁移：复制规则文件到新设备对应目录，即可一键迁移配置

## 2.3 个人规则模板（官方推荐+社区最佳实践）

```markdown
# 个人规则 - 高效开发者配置（通用版）
---always
1.  所有对话、注释、文档必须使用中文，技术术语搭配通俗解释
2.  优先提供Windows 11 + PowerShell解决方案，兼顾macOS适配
3.  技术类回答必须包含：核心原理+分步步骤+可运行代码+验证方法
4.  代码风格：2空格缩进，单引号字符串，箭头函数优先，禁止多余空行
5.  报错处理：优先定位根源，给出修复代码，标注注意事项
6.  回答篇幅：精简干练，重点突出，无关内容不赘述
7.  编码偏好：前端优先React+TypeScript，后端优先Node.js+SpringBoot
---end
```

## 2.4 极简个人规则模板（新手专用）

```markdown
# 个人规则 - 新手极简版
---always
1.  全程中文交互
2.  代码带关键注释
3.  分步讲解，通俗易懂
4.  给出可直接运行的代码
---end
```

# 三、项目规则配置指南（团队协作版）

## 3.1 官方配置步骤

1. 打开目标项目，进入**项目设置** → **规则管理**

2. 点击**+ 创建 project_rules.md**，系统自动生成`.trae/rules`目录

3. 团队共同编写规则，明确强制规范、可选规范、禁忌要求

4. 将规则文件提交至Git，纳入版本管控，同步给所有团队成员

5. 成员拉取代码后，规则自动生效，无需手动配置

6. 修改规则后需重新提交Git，全员同步更新，保证规则统一

## 3.2 项目规则核心写法（官方语法）

- `---always`：标记始终生效的规则，优先级最高，不受上下文影响，多用于项目基础信息

- `---when 条件`：按条件触发规则，适配特定文件、场景

- `!override`：强制覆盖个人规则，保障团队规范落地

- `!include`：引入子规则文件，拆分复杂规则，便于维护

## 3.3 项目规则模板（团队通用完整版）

```markdown
# 项目规则 - XX项目开发规范
## 1. 项目基础信息（强制生效）
---always
项目名称：XX管理系统
技术栈：Java17 + SpringBoot3.2 + Vue3 + TypeScript5 + MySQL8
包管理器：后端Maven，前端pnpm
核心目录：
  后端：src/main/java/com/xx/project
  前端：src/views、src/api
本地启动命令：
  后端：mvn spring-boot:run
  前端：pnpm dev
健康检查：
  后端：http://localhost:8080/actuator/health
  前端：http://localhost:5173
---

## 2. 代码风格规范（强制）
### 后端Java规范
---when file: *.java
1.  缩进4空格，类名PascalCase，方法/变量camelCase，常量全大写下划线分隔
2.  必须添加Javadoc注释，包含@param、@return、@throws
3.  禁止使用魔法值，统一抽取常量
4.  开启全局异常捕获，统一返回格式{"code":200,"message":"success","data":null}
---

### 前端规范
---when file: *.vue,*.ts,*.js
1.  缩进2空格，单引号，组件名PascalCase
2.  使用Tailwind CSS，禁止内联样式
3.  接口请求统一封装，携带请求头，捕获异常
4.  单个文件代码不超过500行，拆分复杂模块
---

## 3. Git协作规范
1.  分支规范：main（生产）、develop（开发）、feature/*（功能）、bugfix/*（修复）
2.  提交信息格式：类型: 描述（feat: 新增用户模块；fix: 修复登录失效）
3.  提交前必须通过代码校验，修复所有ESLint、语法报错
4.  核心代码必须经过Code Review，方可合并

## 4. 安全与质量规范
1.  禁止明文存储密钥、密码，使用环境变量配置
2.  接口必须做权限校验，防止越权操作
3.  防止SQL注入、XSS攻击，做好参数校验
4.  核心代码单元测试覆盖率不低于80%
```

---

## 3.4 项目规则四类生效方式（官方原版）

项目规则可按需配置生效方式，系统会自动修改`alwaysApply`属性，搭配对应参数即可实现精准触发，适配不同场景需求。

|生效方式|核心说明|属性配置|
|---|---|---|
|**始终生效**|规则在项目内所有AI对话中默认生效，无需手动触发，适合核心规范、项目基础信息|alwaysApply自动设为true，无需额外配置|
|**指定文件生效**|仅作用于匹配通配符的文件，提及对应文件时自动启用，适合分语言、分目录规范|alwaysApply自动设为false，配置globs文件匹配规则（如*.js、src/**/*.ts）|
|**智能生效**|AI根据场景相关性判断是否启用，无需手动指定，灵活适配特定开发场景|alwaysApply自动设为false，填写description描述适用场景|
|**手动触发生效**|仅在对话中用#Rule提及规则时生效，优先级最高，适合临时规范、专项要求|alwaysApply自动设为false，无额外属性，对话中手动调用|
## 3.5 对话中引用规则（官方规范）

- **始终生效规则**：对话默认自动引用，输入框左上角会展示已启用的规则标识

- **手动触发规则**：对话输入框中通过`#Rule`直接引用，即可强制启用

- **优先级提示**：#Rule引用优先级最高，指定文件、智能生效规则，也可通过#Rule手动触发，强制在当前对话生效

## 3.6 编辑与删除规则（官方步骤）

1. 进入设置中心，在规则列表里找到目标个人规则/项目规则

2. 点击规则右侧的设置图标，弹出操作菜单

3. 按需选择**编辑**，修改规则内容后保存；或选择**删除**，确认后彻底移除规则

4. 修改规则后，建议开启全新对话使用，避免历史上下文冲突

# 四、官方原版规则示例

以下为官方标准项目规则写法，纯Markdown语法，贴合平台规范，可直接套用修改。

```markdown
# 编码规范
## 通用规范
- 避免不必要的对象复制或克隆
- 避免多层嵌套，提前返回
- 使用适当的并发控制机制

## 代码可读性
### 命名约定
- 使用有意义的、描述性的名称
- 遵循项目或语言的命名规范
避免缩写和单字母变量（除非是约定俗成的，如循环中的 `i`）

### 代码组织
- 相关代码放在一起
- 一个函数只做一件事
- 保持适当的抽象层次

### 注释与文档
- 注释应该解释为什么，而不是做什么
- 为公共 API 提供清晰的文档
- 更新注释以反映代码变化

## 性能优化
### 内存优化
- 避免不必要的对象创建
- 及时释放不再需要的资源
- 注意内存泄漏问题

### 计算优化
- 避免重复计算
- 使用适当的数据结构和算法
- 延迟计算直到必要时

### 并行优化
- 识别可并行化的任务
- 避免不必要的同步
- 注意线程安全问题
```

# 五、高级配置技巧（官方权威+进阶玩法）

**优先级排序（官方定义）**：项目规则（带always强制标记） > 普通项目规则 > 个人规则 > 软件默认规则

- 规则内容冲突时，高优先级规则会完全覆盖低优先级规则，不会叠加生效

- 团队核心强制规范，必须添加always标记，杜绝个人习惯干扰团队标准

## 5.1 规则高级语法与属性说明

|语法/属性|功能作用|使用须知|
|---|---|---|
|alwaysApply|控制规则是否全局默认生效|始终生效的规则，系统自动设为true；其余生效方式自动设为false|
|globs|文件匹配规则，限定作用范围|支持通配符，多条匹配规则用逗号分隔，必须使用项目相对路径|
|description|描述适用场景，供AI智能判定|语句直白具体，写明适用场景、约束要求、执行标准|
|#Rule|手动触发规则，优先级置顶|对话中直接输入，可强制启用非默认生效的规则|
|---always|标记规则始终生效，不受上下文干扰|多用于项目基础信息、核心强制规范|
|---when|按条件触发，精准适配场景|配合文件后缀、目录使用，格式固定---when file: 匹配规则|
|!override|强制覆盖个人规则|团队规范必备，杜绝个性化偏差|
|!include|引入外部子规则|拆分大型规则，模块化管理|
## 5.2 规则编写黄金准则（官方最佳实践）

1. **严控粒度**：单条规则只包含一个要求，内容聚焦不堆砌，清晰易懂，便于AI识别执行

2. **杜绝冲突**：各条规则之间不得相互矛盾、相互覆盖，防止AI执行混乱、输出偏差

3. **路径规范**：涉及文件、目录，一律使用相对于项目根目录的相对路径，保证定位精准

4. **相关性优先**：引用规则时，只选用和当前任务、对话强相关的规则，不冗余加载

5. **新旧隔离**：新建或修改规则后，必须开启全新对话使用，避免历史上下文干扰规则生效

6. **存量代码适配**：项目旧代码不规范时，需明确告知AI执行重构，强制遵循新规则

7. **简洁明确**：单条规则只讲一件事，不用模糊词汇（不用“适当缩进”，用“2空格缩进”）

8. **无冲突**：规则之间不能互相矛盾，避免AI识别混乱

9. **可执行**：规则可验证、可检查，不是空泛口号

10. **最小化**：保留核心规则，删减冗余内容，防止规则过多导致AI失效

11. **易维护**：分类清晰，模块化编写，便于后期修改更新

# 六、权限管控与团队管理（实战指南）

## 6.1 规则权限分级

- **管理员/技术架构师**：拥有规则创建、修改、审核、发布全权，把控整体规范方向

- **项目负责人**：负责日常规则维护、更新同步，督促团队成员执行规范

- **普通开发者**：仅可查看、使用规则，无修改权限，严格遵循既定规范开展工作

## 6.2 团队规则同步方案

1. 将`.trae/rules`目录完整纳入Git版本控制，禁止加入gitignore忽略

2. 设置分支保护规则，规则文件修改需经过审核方可合并，防止私自篡改

3. 新人入职，拉取项目代码后自动加载规则，无需手动配置，快速对齐标准

4. 定期更新规则，同步团队全员，统一执行口径，避免标准不一

## 6.3 CI/CD自动化校验

可将规则校验接入CI/CD流程（GitHub Actions、GitLab CI、Jenkins），代码提交时自动检查是否符合规范，不合格代码禁止合并：

- 配置自动化脚本，校验代码格式、命名规范、注释完整性

- 结合ESLint、CheckStyle等工具，和项目规则联动校验

- 违规自动拦截，给出修改提示，提升团队代码质量

## 6.4 兼容配置文件（官方专属）

### 6.4.1 AGENTS.md 配置

存放于项目根目录的轻量级规则文件，属于项目级规范，用于约束AI智能体行为，支持跨IDE复用。

- 生效方式：进入**设置>规则**，开启**将AGENTS.md包含在上下文中**（默认开启）

- 适用场景：定制智能体工作流程、专项任务约束、自动化开发规范

### 6.4.2 CLAUDE.md与CLAUDE.local.md

TRAE全面兼容Claude规则体系，导入含此类文件的项目，规则自动迁移，无需重写。

- 生效方式：进入**设置>规则**，开启**将CLAUDE.md包含在上下文中**

- 适用范围：项目级规范，与原生项目规则共用优先级，适配跨平台协作

# 七、配置验证与故障排查（官方+实操）

## 7.1 规则生效验证步骤

1. **命令自检**：Trae CN终端输入`trae rules check`，一键排查语法错误、规则冲突

2. **对话实测**：关闭旧对话，新建空白对话，下达指令核查输出是否合规

3. **文件核验**：核对文件名、存储路径，杜绝拼写错误、编码异常

4. **日志核查**：快捷键Ctrl+Shift+L打开日志面板，搜索rules关键词，查看加载状态

## 7.2 常见问题及解决方案

|问题现象|排查方向|解决办法|
|---|---|---|
|规则完全不生效|路径错误、文件名写错、文件编码异常、缓存未清空|修正路径文件名，改用UTF-8编码，开启新对话测试|
|个人规则与项目规则冲突|项目规则优先级不足，无强制覆盖标记|项目规则添加---always或!override强制生效|
|团队成员规则不一致|未拉取最新代码、本地私自修改规则|拉取最新Git代码，禁止本地修改项目规则文件|
|AI识别错乱、输出跑偏|规则冗余、表述模糊、条款矛盾|精简规则，简化语句，删除冲突内容|
|规则语法报错|高级语法使用失误，格式不闭合|修正标记格式，补齐---闭合符，用官方命令自检|
## 7.3 快速排错步骤

1. 重启Trae CN软件，刷新配置缓存

2. 备份规则文件，清空内容，逐段添加测试，定位问题规则

3. 核对文件名、目录名，杜绝拼写错误

4. 降级规则复杂度，先用极简规则测试生效状态
