DEC-001: API Gateway 高可用方案选型

决策日期： 2025-01-15
决策人： 架构评审委员会
状态： ✅ 已决策

背景

当前 API Gateway 存在单点故障风险，一旦 Gateway 服务宕机，整个平台将无法访问。需要设计高可用方案消除此风险。

候选方案

方案 描述 优点 缺点
A: Nginx + Keepalived Nginx 做七层负载均衡 + Keepalived 做 VIP 漂移 成熟稳定、社区活跃、配置简单 需要额外服务器
B: 云负载均衡器 (SLB/ALB) 使用云厂商提供的负载均衡服务 托管服务、无需运维 绑定云厂商、成本较高
C: Consul + Fabio 服务发现 + 动态路由 动态感知、适合微服务 配置复杂、额外组件
D: Traefik 原生支持 K8s 的反向代理 K8s 集成好 学习曲线、文档较少

评估矩阵

评估维度 权重 Nginx+Keepalived 云 SLB Consul+Fabio Traefik
可靠性 30% 5 5 4 4
易运维性 25% 4 5 2 3
成本 20% 3 2 4 4
性能 15% 5 4 4 3
扩展性 10% 4 3 5 4
加权总分 4.30 3.95 3.50 3.55

决策结论

选择方案 A: Nginx + Keepalived

决策依据

  1. 成熟稳定：Nginx 是经过大规模验证的反向代理，Keepalived 在 LVS 场景有多年应用
  2. 性能优秀：Nginx 基于 epoll，单机可处理 10W+ 并发
  3. 成本可控：使用开源组件，无需额外license费用
  4. 运维简单：配置格式清晰，团队已有 Nginx 使用经验
  5. 可回滚：方案切换对业务透明，支持快速回退

权衡考量

考量点 说明
资源投入 需要 2 台独立服务器用于 Nginx，但可复用现有服务器
复杂度 比纯云方案复杂度略高，但远低于 Consul 等服务发现方案
灵活性 静态配置，变更需要修改配置并 reload；可接受

放弃的方案

放弃方案 原因
云 SLB 成本较高（按流量/带宽计费），且增加云厂商绑定
Consul + Fabio 引入过多组件，增加运维复杂度和故障点
Traefik 与现有 Kubernetes 部署模式整合需要额外工作

后续关注点

  • [ ] 监控 Nginx 连接数和内存使用
  • [ ] 考虑后续迁移到更现代的方案（如 Envoy）
  • [ ] 评估 Keepalived 单点风险（VIP 漂移依赖网络）

DEC-002: Kafka 消息可靠性配置策略

决策日期： 2025-01-15
决策人： 后端架构组
状态： ✅ 已决策

背景

当前 Kafka 配置使用默认参数，存在消息丢失和重复消费风险。需要优化配置确保消息可靠性。

候选方案

方案 描述 可靠性 性能影响
A: 高可靠模式 acks=all, retries=3, idempotence=true 最高 延迟增加 30-50%
B: 中等可靠 acks=1, retries=3 中等 延迟增加 10-20%
C: 高性能模式 acks=0, async send 最低 几乎无影响
D: 可配置模式 根据 Topic 类型动态配置 可控 取决于配置

决策结论

选择方案 A + D 混合：核心业务使用高可靠模式，非核心业务可降级

决策依据

  1. 业务重要性：订单、支付等核心业务消息不能丢失
  2. 性能可接受：Kafka 在同一数据中心延迟通常 < 10ms，配置调整后 < 50ms 可接受
  3. 社区最佳实践：Kafka 官方推荐生产环境使用 acks=all
  4. 幂等性支持：开启 idempotence 可避免生产者重试导致的消息重复

配置规范

# Kafka 配置规范

# 核心业务 (order, payment)
core_topics:
  acks: all
  retries: 5
  idempotence: true
  replication_factor: 3
  min_insync_replicas: 2

# 普通业务
normal_topics:
  acks: all
  retries: 3
  idempotence: true
  replication_factor: 3
  min_insync_replicas: 2

# 非核心业务 (日志, 埋点)
non_core_topics:
  acks: 1
  retries: 3
  replication_factor: 3
  min_insync_replicas: 2

权衡考量

考量点 说明
延迟增加 高可靠模式会增加端到端延迟，但控制在可接受范围
吞吐量下降 约下降 20-30%，需要评估容量
运维复杂度 引入死信队列增加运维负担

后续关注点

  • [ ] 监控 producer 发送成功率和重试率
  • [ ] 定期检查死信队列积压情况
  • [ ] 评估消息处理幂等性的实现质量
  • [ ] 考虑引入 Kafka Streams 实现 Exactly-Once

DEC-003: 数据库双写一致性方案选型

决策日期： 2025-01-15
决策人： 后端架构组 + DBA
状态： ✅ 已决策

背景

系统使用 MySQL + Redis 进行数据存储，读操作需要同时查询数据库和缓存。当前存在缓存与数据库不一致的风险。

候选方案

方案 描述 一致性 复杂度 性能
A: Cache Aside 读时填充，写时删除缓存 最终一致 低 高
B: 延迟双删 Cache Aside + 延迟再次删除 最终一致 中 中
C: 订阅 binlog 监听数据库变更同步缓存 准实时 高 高
D: 分布式事务 两阶段提交/Seata 强一致 很高 低

评估矩阵

评估维度 权重 Cache Aside 延迟双删 binlog 分布式事务
一致性 30% 3 4 5 5
复杂度 25% 5 4 2 1
性能 20% 5 4 4 2
可运维 15% 5 4 2 1
侵入性 10% 5 4 2 1
加权总分 4.30 3.85 3.00 2.15

决策结论

选择方案 B: 延迟双删策略 + 补偿机制

决策依据

  1. 团队能力：分布式事务方案需要较深的技术积累，团队目前不具备
  2. 业务需求：大部分场景允许短暂不一致（< 1秒），无需强一致
  3. 复杂度可控：延迟双删方案代码侵入性小，团队可自主实现
  4. 可监控：可建立数据一致性检查任务，发现不一致及时修复

延迟双删实现要点

DEC-004: 服务熔断器技术选型

决策日期： 2025-01-15
决策人： 后端架构组
状态： ✅ 已决策

背景

微服务架构中存在服务间调用，当前缺乏熔断机制，一个服务故障可能导致级联失败。需要引入熔断器实现故障隔离。

候选方案

方案 描述 GitHub Stars 最后更新 维护状态
A: opossum Node.js 熔断器 1.5k+ 2024 活跃
B: circuit-breaker-js 轻量级熔断器 200+ 2022 低活跃
C: Hystrix (JS) Java Hystrix 的 JS 移植 500+ 2021 低活跃
D: 自研 基于 Map 实现 - - 完全可控

评估矩阵

评估维度 权重 opossum circuit-breaker-js Hystrix-JS 自研
功能完整性 25% 5 3 4 2
稳定性 25% 5 4 3 1
文档质量 15% 5 3 3 5
性能 15% 4 5 3 5
社区活跃度 10% 5 2 1 0
学习成本 10% 4 4 3 2
加权总分 4.65 3.55 3.00 2.10

决策结论

选择方案 A: opossum

决策依据

  1. 功能完整：支持熔断、降级、重试、超时、限流等完整能力
  2. 稳定性高：经过生产环境验证，有 1.5k+ GitHub Stars
  3. 文档完善：提供完整的文档和示例
  4. 性能良好：基于 Promise实现，对性能影响小
  5. 状态可视：提供熔断器状态事件，便于监控

核心配置示例

const CircuitBreaker = require('opossum');

// 创建熔断器
const breaker = new CircuitBreaker(myAsyncFunction, {
  // 超时时间（ms）
  timeout: 3000,

  // 错误率阈值（%）
  errorThresholdPercentage: 50,

  // 熔断持续时间（ms）
  resetTimeout: 30000,

  // 最小请求数
  volumeThreshold: 10,
});

// 设置降级函数
breaker.fallback(() => ({ fallback: true }));

// 使用
breaker.fire(params).then(result => {
  console.log('Success:', result);
}).catch(error => {
  console.error('Error:', error);
});

权衡考量

考量点 说明
依赖引入 引入第三方库，但经过社区验证
定制化 基本满足需求，复杂定制需深入源码
监控 提供事件，需自行对接监控系统

DEC-006: 缓存中间件选型

决策日期： 2025-01-15
决策人： 后端架构组
状态： ✅ 已决策

背景

系统需要缓存层提升读取性能，需要选择合适的缓存中间件。

候选方案

方案 描述 一致性 性能 功能
A: Redis 内存数据库 单机强一致 极高 丰富
B: Memcached 纯内存缓存 无 高 简单
C: Redis Cluster Redis 集群模式 最终一致 高 丰富
D: Dragonwell 阿里巴巴 JDK - - -

决策结论

选择方案 C: Redis Cluster

决策依据

  1. 功能丰富：支持 String、Hash、List、Set、ZSet 等数据结构
  2. 高可用：Cluster 模式支持自动故障转移
  3. 水平扩展：可动态增加节点分担负载
  4. 持久化：支持 RDB + AOF 混合持久化
  5. 生态完善：Node.js 社区有成熟的客户端（ioredis）

权衡考量

考量点 说明
运维复杂度 Cluster 比单机运维复杂
一致性 Cluster 模式下不是强一致，存在小窗口不一致
成本 内存消耗较大，需要合理规划

后续关注点

  • [ ] 规划 Redis Cluster 节点数量和规格
  • [ ] 配置合理的持久化策略
  • [ ] 监控内存使用率和 key 大小
  • [ ] 评估 Redis 7.0 新特性
